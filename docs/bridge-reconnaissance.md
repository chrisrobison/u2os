## 1. LARC Cross-Process API (U2OS)

### What is actually implemented
- In `u2os/core`, server-side events are implemented by [`core/eventBus.js`](/Users/cdr/Projects/u2os/core/eventBus.js), not by an external LARC package.
- `@larcjs/core-lite` appears in the browser runtime (`ui/app/index.html`) for client orchestration, but there is no server-side cross-process transport in `core/`, `modules/`, or `core-modules/`.

### How subscriptions are established in U2OS
- Capability packages are loaded in-process by [`core/pluginLoader.js`](/Users/cdr/Projects/u2os/core/pluginLoader.js).
- `events.js` files call `eventBus.subscribe(...)` during module initialization.
- `pluginLoader` wraps the bus with a permission-checked proxy (`createModuleEventBusProxy`) and forwards calls to the same in-memory `EventBus` instance.

### Exact publish/subscribe signatures
From [`core/eventBus.js`](/Users/cdr/Projects/u2os/core/eventBus.js):
- `subscribe(eventName, handler)`
  - Registers `handler` for `eventName`.
  - Returns an unsubscribe function: `() => void`.
- `async publish(eventName, payload = {})`
  - Persists first (if `persistEvent` is configured), then dispatches to handlers.
- Handler input shape:
  - `handler({ eventName, payload, emittedAt })`

From [`core/pluginLoader.js`](/Users/cdr/Projects/u2os/core/pluginLoader.js) module proxy:
- `subscribe(eventName, handler)` (permission-checked, then forwarded)
- `publish(eventName, payload = {})` (permission-checked, then forwarded)

### Connection loss / reconnection behavior
- No cross-process connection is used, so no connection-loss protocol exists.
- Handlers are process-memory state (`Map` in `EventBus`); on process restart/crash they are rebuilt only when modules are reloaded.
- No replay/reconnect handshake is implemented at the bus layer.

### Namespacing / wildcard support (cross-process)
- Event names are free-form strings (dot notation by convention).
- Wildcard support is only exact `'*'` subscriber matching all events (`EventBus` merges exact + `'*'` handlers).
- No prefix wildcards (e.g., `transportation.*`) and no namespace isolation semantics are implemented.

### Naming conventions observed
- Lowercase dot notation (`domain.entity.action`).
- Generic CRUD events are generated by `entity.slice(0, -1)` in `core/router.js`.
- Domain-prefixed module events exist (`transportation.*`, `appointment.booked`).

### Naming anomalies observed
- Singularization by `slice(0,-1)` creates malformed names for some entities:
  - `pet` -> `pe.*`
  - `transportation_addresses` -> `transportation_addresse.*`
  - `transportation_buses` -> `transportation_buse.*`
- `clamp.created` is emitted from two different routes with two different payload shapes.
- `payment.received` is subscribed to in modules, but no publisher exists in `core/`, `modules/`, or `core-modules/`.

## 4. Integration Surface Analysis

### U2OS events that are natural triggers for MindGraph workflows
- `customer.created` / `customer.updated`
  - Trigger customer onboarding, enrichment, or follow-up workflows.
- `appointment.booked`
  - Trigger reminder generation, staffing checks, and pre-visit prep workflows.
- `order.created`
  - Trigger fulfillment/approval workflows.
- `invoice.created` and `payment.created` (or future `payment.received`)
  - Trigger billing reconciliation and collections workflows.
- `task.created` / `task.updated`
  - Trigger downstream operational automations.
- `transportation.request.created`
  - Trigger dispatch planning and route validation workflows.
- `transportation.trip.scheduled` / `transportation.trip.started` / `transportation.trip.completed`
  - Trigger runtime route monitoring, completion QA, and billing handoff workflows.

### MindGraph action-completion events that should write back to U2OS
- `runtime.agent.run.completed`
  - Write completion result summary to U2OS (`events` table and/or domain record notes/status).
- `runtime.agent.run.failed`
  - Write failure incident and create/raise U2OS task for human follow-up.
- `runtime.run.cancelled`
  - Write cancellation status to active U2OS operation/task.
- `runtime.run.history.appended`
  - Append audit trail entries in U2OS for traceability.
- `runtime.data.refreshed`
  - Persist source freshness timestamps / diagnostics back to U2OS data-health records.
- `runtime.error.appended`
  - Persist error telemetry to U2OS operational logging/audit entities.

### Gaps: U2OS operations with no corresponding event yet (recommended additions)
- `payment.received` publisher is missing (despite active subscribers).
- No tenant/control-plane lifecycle events (customer/instance/domain create/update/status).
- No onboarding/settings mutation events from `core/routes/system.js` updates.
- No auth/login/security audit events emitted on key auth flows.
- Transportation lifecycle lacks explicit cancel/reschedule/no-show events.
- Generic CRUD event naming is not schema-safe for irregular plurals.
- Event payloads do not include a standard envelope (`tenantId`, `correlationId`, version).

### Gaps: MindGraph action types with no U2OS equivalent yet
- Graph-edit lifecycle (`graph.node.*`, `graph.edge.*`, document undo/redo/load/save) has no U2OS workflow-graph domain equivalents.
- Runtime execution control (`runtime.*.requested`, cancel semantics) lacks U2OS workflow command event counterparts.
- Runtime observability streams (`runtime.trace.appended`, `task.queue.updated`) lack first-class U2OS workflow telemetry entities/events.
- UI/runtime config events (`ui.settings.*`, `runtime.provider.settings.*`) have no U2OS settings event analog.

## 5. Recommended Bridge Event Naming Convention

### Proposed schema
- `bridge.<domain>.<resource>.<action>.<phase>.v1`

Where:
- `domain`: `crm`, `scheduling`, `billing`, `transport`, `workflow`, `ops`
- `resource`: business noun (`customer`, `appointment`, `invoice`, `payment`, `trip`, `run`, ...)
- `action`: verb (`create`, `update`, `book`, `schedule`, `start`, `complete`, `execute`, ...)
- `phase`: `requested`, `started`, `completed`, `failed`, `changed`
- version suffix fixed in name (`v1`) and repeated in payload metadata

### Recommended payload envelope
```json
{
  "eventName": "bridge.billing.payment.receive.completed.v1",
  "schemaVersion": "v1",
  "sourceSystem": "u2os",
  "tenantId": "<tenant-id>",
  "correlationId": "<trace-or-workflow-id>",
  "causationId": "<upstream-event-id>",
  "occurredAt": "2026-04-08T00:00:00.000Z",
  "payload": {}
}
```
